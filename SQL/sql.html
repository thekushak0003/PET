Basic

Introduction to SQL & Command Types

SQL (Structured Query Language) is the standard language for managing relational databases. Understanding how SQL commands are categorized is the very first topic in any placement test.

What is a Relational Database?

A Relational Database Management System (RDBMS) stores data in structured tables (relations) made of rows and columns. Examples: MySQL, PostgreSQL, Oracle, MS SQL Server, SQLite.

  • Table (Relation): A collection of rows and columns
  • Row (Tuple): A single record in a table
  • Column (Attribute): A field in a table
  • Primary Key: Uniquely identifies each row
  • Foreign Key: Links rows between two tables

SQL Command Categories

Category Full Name Commands Purpose
DDL Data Definition Language CREATE, ALTER, DROP, TRUNCATE, RENAME Define/modify schema structure
DML Data Manipulation Language INSERT, UPDATE, DELETE, SELECT* Manipulate data inside tables
DQL Data Query Language SELECT Retrieve data (sometimes part of DML)
DCL Data Control Language GRANT, REVOKE Access permissions
TCL Transaction Control Language COMMIT, ROLLBACK, SAVEPOINT Manage transactions
💡 Placement tip: DDL changes are auto-committed; DML changes need an explicit COMMIT (in most RDBMS). DDL operates on schema, DML on data.

Order of SQL Query Execution

This is a very common placement question. Queries are written in one order but executed in another:

Written order vs Execution order
-- Written order (how you type it):
SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT

-- Logical execution order:
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
⚠️ Because SELECT runs after WHERE, you cannot use a column alias in a WHERE clause. You can use it in ORDER BY (which runs after SELECT).

SQL Data Types

Choosing the right data type is fundamental to database design and a common interview topic.

Numeric Types

Type Range / Size Use Case
TINYINT -128 to 127 Flags, small codes
SMALLINT -32,768 to 32,767 Small counters
INT / INTEGER -2B to 2B (4 bytes) Most IDs and counts
BIGINT ±9.2 × 10¹⁸ Large PKs, timestamps
DECIMAL(p,s) Exact fixed precision Money, measurements
FLOAT / REAL Approximate 4-byte Scientific, approximate
DOUBLE Approximate 8-byte High-range floats

String / Text Types

Type Storage When to use
CHAR(n) Fixed n bytes Fixed-length: country codes, phone numbers
VARCHAR(n) Variable, max n bytes Names, emails — variable length
TEXT Large variable Long descriptions, articles
ENUM('a','b') 1-2 bytes Fixed set of values: gender, status
💡 CHAR vs VARCHAR: CHAR pads with spaces to fill width (faster for fixed-length). VARCHAR stores only actual data + length prefix (saves space). Use CHAR for fixed-length fields like PIN codes.

Date & Time Types

TIME HH:MM:SS Time of day
DATETIME YYYY-MM-DD HH:MM:SS Timestamps (no timezone)
TIMESTAMP YYYY-MM-DD HH:MM:SS Auto-updating records (stores UTC)

DDL Commands

Data Definition Language commands define and modify the structure (schema) of the database. These are auto-committed — they cannot be rolled back.

CREATE TABLE

SQL
CREATE TABLE employees (
    emp_id      INT           PRIMARY KEY AUTO_INCREMENT,
    name        VARCHAR(100)  NOT NULL,
    email       VARCHAR(150)  UNIQUE,
    salary      DECIMAL(10,2) DEFAULT 30000.00,
    dept_id     INT,
    hire_date   DATE,
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);

ALTER TABLE

SQL
-- Add a new column
ALTER TABLE employees ADD COLUMN phone VARCHAR(15);

-- Modify column type/size
ALTER TABLE employees MODIFY COLUMN name VARCHAR(200);

-- Rename a column (MySQL 8+)
ALTER TABLE employees RENAME COLUMN phone TO mobile;

-- Drop a column
ALTER TABLE employees DROP COLUMN mobile;

-- Add a constraint
ALTER TABLE employees ADD CONSTRAINT chk_salary CHECK (salary > 0);

DROP vs TRUNCATE vs DELETE

Command Removes Structure? Removes Data? Can Rollback? Triggers Fired? Speed
DROP TABLE Yes (entire table) Yes No (DDL) No Fast
TRUNCATE No (keeps table) Yes (all rows) No (DDL) No Fast
DELETE No Yes (selective) Yes (DML) Yes Slower
⚠️ Placement favourite: DROP removes the table + data + structure. TRUNCATE removes all rows but keeps table. DELETE removes rows and supports WHERE clause.
SQL
DROP TABLE employees;        -- removes table completely
TRUNCATE TABLE employees;   -- removes all data, table stays
DELETE FROM employees;       -- removes all rows (slow, DML)
DELETE FROM employees WHERE emp_id = 5;  -- selective delete

DML Commands — INSERT, UPDATE, DELETE

Data Manipulation Language modifies the actual data inside tables. These operations can be rolled back within a transaction.

INSERT

SQL
-- Insert a single row
INSERT INTO employees (name, email, salary, dept_id)
VALUES ('Arjun Sharma', 'arjun@company.com', 55000, 3);

-- Insert multiple rows at once
INSERT INTO employees (name, salary) VALUES
    ('Priya Patel',  62000),
    ('Rahul Singh',  48000),
    ('Neha Gupta',   71000);

-- Insert from another table (INSERT INTO ... SELECT)
INSERT INTO emp_archive
SELECT * FROM employees WHERE hire_date < '2020-01-01';

UPDATE

SQL
-- Update specific rows
UPDATE employees
SET salary = salary * 1.10,  -- 10% raise
    email  = 'new@company.com'
WHERE dept_id = 2;

-- Update with JOIN (MySQL syntax)
UPDATE employees e
JOIN departments d ON e.dept_id = d.dept_id
SET e.salary = e.salary * 1.15
WHERE d.dept_name = 'Engineering';
⚠️ Always use WHERE in UPDATE and DELETE. Without it, every row gets modified/deleted!

DELETE

SQL
-- Delete specific rows
DELETE FROM employees WHERE emp_id = 42;

-- Delete with subquery
DELETE FROM employees
WHERE dept_id IN (
    SELECT dept_id FROM departments WHERE location = 'Mumbai'
);

SELECT, Filtering & Sorting

The SELECT statement is the heart of SQL. Master every clause — these appear in almost every placement test.

Basic SELECT Syntax

SQL
SELECT column1, column2      -- choose columns (* = all)
FROM   table_name            -- source table
WHERE  condition             -- filter rows
GROUP BY column1             -- group for aggregation
HAVING aggregate_condition  -- filter groups
ORDER BY column1 DESC       -- sort result
LIMIT  10 OFFSET 20;         -- pagination

WHERE Clause Operators

Operator Example Meaning
= salary = 50000 Equal
<> or != dept_id != 3 Not equal
<, >, <=, >= salary >= 40000 Comparison
BETWEEN…AND salary BETWEEN 40000 AND 80000 Inclusive range
IN (…) dept_id IN (1, 2, 3) Match any in list
NOT IN (…) dept_id NOT IN (4, 5) Exclude list values
LIKE name LIKE 'A%' Pattern match (% = any, _ = one char)
IS NULL email IS NULL Check for NULL
IS NOT NULL email IS NOT NULL Non-NULL check
AND / OR / NOT salary > 50k AND dept=2 Logical operators

LIKE Pattern Matching

SQL
SELECT * FROM employees WHERE name LIKE 'A%';      -- starts with A
SELECT * FROM employees WHERE name LIKE '%kumar';   -- ends with kumar
SELECT * FROM employees WHERE name LIKE '%raj%';    -- contains raj
SELECT * FROM employees WHERE name LIKE '_a%';      -- 2nd char is 'a'
SELECT * FROM employees WHERE name LIKE '___';      -- exactly 3 chars

ORDER BY & DISTINCT

SQL
SELECT DISTINCT dept_id FROM employees;   -- unique values only

SELECT name, salary FROM employees
ORDER BY salary DESC,   -- sort by salary high→low
         name   ASC;    -- then by name A→Z

-- LIMIT and OFFSET for pagination
SELECT * FROM employees ORDER BY emp_id LIMIT 10 OFFSET 20;
-- Returns rows 21-30

Aliases

SQL
-- Column alias
SELECT emp_id AS id,
       CONCAT(first_name, ' ', last_name) AS full_name,
       salary * 12 AS annual_salary
FROM employees;

-- Table alias (useful in JOINs and subqueries)
SELECT e.name, d.dept_name
FROM employees e, departments d
WHERE e.dept_id = d.dept_id;

Constraints

Constraints enforce rules on data in tables, ensuring data integrity. A very frequent topic in theory-based placement rounds.

Constraint Purpose Key Rules
PRIMARY KEY Uniquely identifies a row NOT NULL + UNIQUE; one per table
FOREIGN KEY Links to PK of another table Enforces referential integrity
UNIQUE All values must be distinct Allows one NULL per column
NOT NULL Disallows NULL values Must always have a value
CHECK Validates value against condition e.g., salary > 0
DEFAULT Sets default value if none given Applied on INSERT
SQL — Constraint Examples
CREATE TABLE orders (
    order_id   INT PRIMARY KEY,
    customer_id INT NOT NULL,
    order_date DATE DEFAULT (CURRENT_DATE),
    amount     DECIMAL(10,2) CHECK (amount > 0),
    status     VARCHAR(20) DEFAULT 'pending',
    CONSTRAINT fk_customer
        FOREIGN KEY (customer_id) REFERENCES customers(id)
        ON DELETE CASCADE
        ON UPDATE SET NULL
);

Foreign Key Actions

Action On DELETE/UPDATE of Parent
CASCADE Delete/update child rows too
SET NULL Set child FK to NULL
SET DEFAULT Set child FK to default value
RESTRICT Prevent the operation if child rows exist
NO ACTION Same as RESTRICT (checked after statement)

Composite Primary Key

SQL
-- When a single column is not enough to uniquely identify a row
CREATE TABLE enrollment (
    student_id INT,
    course_id  INT,
    grade      CHAR(1),
    PRIMARY KEY (student_id, course_id)  -- composite PK
);

Aggregate Functions & GROUP BY

Aggregate functions operate on sets of rows to produce a single value. GROUP BY groups rows to apply aggregates per group.

Core Aggregate Functions

SQL
SELECT
    COUNT(*)              AS total_rows,       -- counts all rows
    COUNT(email)         AS non_null_emails,  -- counts non-NULL
    COUNT(DISTINCT dept_id) AS dept_count,    -- distinct depts
    SUM(salary)          AS total_payroll,
    AVG(salary)          AS avg_salary,
    MIN(salary)          AS min_salary,
    MAX(salary)          AS max_salary
FROM employees;

GROUP BY

SQL
-- Aggregate per group
SELECT
    dept_id,
    COUNT(*)         AS headcount,
    AVG(salary)     AS avg_salary,
    MAX(salary)     AS highest
FROM employees
GROUP BY dept_id
ORDER BY avg_salary DESC;
💡 Rule: Every column in SELECT must either be in GROUP BY or wrapped in an aggregate function.

WHERE vs HAVING

SQL
-- WHERE filters ROWS (before grouping)
-- HAVING filters GROUPS (after grouping)

SELECT dept_id, AVG(salary) AS avg_sal
FROM employees
WHERE hire_date >= '2020-01-01'   -- filter rows first
GROUP BY dept_id
HAVING AVG(salary) > 50000        -- filter groups after
ORDER BY avg_sal DESC;
⚠️ Cannot use column alias in HAVING (in most databases). Use the full expression: HAVING AVG(salary) > 50000, not HAVING avg_sal > 50000.

NULL Handling

NULL means "unknown" or "absent" — it is NOT zero, NOT empty string. NULL comparisons behave unexpectedly and are a classic placement trap.

NULL Comparison Traps

SQL
-- These ALL return FALSE/unknown, NOT what you might expect
SELECT * FROM emp WHERE email = NULL;     -- WRONG ❌
SELECT * FROM emp WHERE email != NULL;    -- WRONG ❌

-- Correct way:
SELECT * FROM emp WHERE email IS NULL;    -- CORRECT ✅
SELECT * FROM emp WHERE email IS NOT NULL; -- CORRECT ✅

NULL Handling Functions

SQL
-- COALESCE: returns first non-NULL value
SELECT COALESCE(phone, email, 'No contact') AS contact
FROM employees;

-- IFNULL (MySQL): replace NULL with a value
SELECT IFNULL(bonus, 0) AS bonus FROM employees;

-- NULLIF: returns NULL if two values are equal
SELECT NULLIF(score, 0) FROM results;
-- Returns NULL if score=0, else returns score (avoids divide-by-zero)

-- Divide-by-zero safe pattern
SELECT total / NULLIF(count_val, 0) AS avg FROM stats;
💡 NULLs and aggregates: COUNT(*) counts ALL rows. COUNT(column) ignores NULLs. SUM/AVG/MIN/MAX all ignore NULLs automatically.

JOINs — The Core of SQL

JOINs combine rows from two or more tables based on a related column. Understanding every type of JOIN is essential for any placement test.

Sample Tables Used in Examples

employees

emp_id | name | dept_id
1 | Alice | 10
2 | Bob | 20
3 | Carol | 30
4 | Dave | NULL

departments

dept_id | dept_name
10 | HR
20 | Engineering
40 | Finance

INNER JOIN

Returns rows that have matching values in BOTH tables.

SQL
SELECT e.name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;
-- Result: Alice/HR, Bob/Engineering  (Carol dept 30 has no match; Dave has NULL)

LEFT JOIN (LEFT OUTER JOIN)

Returns ALL rows from the left table, matched rows from right. Non-matched right columns become NULL.

SQL
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
-- Result: Alice/HR, Bob/Engineering, Carol/NULL, Dave/NULL

-- Find employees with NO department (classic LEFT JOIN trick)
SELECT e.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id
WHERE d.dept_id IS NULL;

RIGHT JOIN

Returns ALL rows from the right table. The mirror of LEFT JOIN.

SQL
SELECT e.name, d.dept_name
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.dept_id;
-- Result: Alice/HR, Bob/Engineering, NULL/Finance

FULL OUTER JOIN

Returns ALL rows from both tables, with NULLs where there's no match. (MySQL doesn't natively support this — use UNION of LEFT + RIGHT JOIN.)

SQL
-- PostgreSQL / SQL Server syntax:
SELECT e.name, d.dept_name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.dept_id;

-- MySQL workaround using UNION:
SELECT e.name, d.dept_name FROM employees e LEFT JOIN departments d ON e.dept_id=d.dept_id
UNION
SELECT e.name, d.dept_name FROM employees e RIGHT JOIN departments d ON e.dept_id=d.dept_id;

CROSS JOIN

Returns the Cartesian product — every row of table A paired with every row of table B. No ON clause needed.

SQL
SELECT e.name, d.dept_name
FROM employees e
CROSS JOIN departments d;
-- Result: 4 employees × 3 departments = 12 rows

SELF JOIN

Joining a table with itself. Used for hierarchical data like org charts (employee–manager relationships).

SQL
-- Find each employee and their manager's name
SELECT
    e.name         AS employee,
    m.name         AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;
✅ Quick memory trick: INNER = intersection | LEFT = left + intersection | RIGHT = right + intersection | FULL = everything | CROSS = all combinations
Medium

Subqueries

A subquery is a query nested inside another query. They can appear in SELECT, FROM, WHERE, and HAVING clauses.

Scalar Subquery (returns one value)

SQL
-- Find employees earning above the company average
SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- Scalar in SELECT clause
SELECT
    name,
    salary,
    (SELECT AVG(salary) FROM employees) AS company_avg
FROM employees;

IN / NOT IN with Subquery

SQL
-- Employees in departments located in Delhi
SELECT name FROM employees
WHERE dept_id IN (
    SELECT dept_id FROM departments
    WHERE location = 'Delhi'
);

Correlated Subquery

A correlated subquery references columns from the outer query. It runs once for each row of the outer query — can be slow on large datasets.

SQL
-- Find the employee with the highest salary in each department
SELECT name, salary, dept_id
FROM employees e1
WHERE salary = (
    SELECT MAX(salary)
    FROM employees e2
    WHERE e2.dept_id = e1.dept_id  -- references outer query
);

EXISTS / NOT EXISTS

SQL
-- EXISTS: returns TRUE if subquery returns any rows
SELECT name FROM employees e
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.emp_id = e.emp_id
);
-- Employees who have placed at least one order

-- NOT EXISTS: employees with NO orders
SELECT name FROM employees e
WHERE NOT EXISTS (
    SELECT 1 FROM orders o
    WHERE o.emp_id = e.emp_id
);

Subquery in FROM (Derived Table)

SQL
-- Using a subquery as a derived table (inline view)
SELECT dept_stats.dept_id, dept_stats.avg_sal
FROM (
    SELECT dept_id, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY dept_id
) dept_stats
WHERE dept_stats.avg_sal > 60000;
💡 EXISTS vs IN: EXISTS stops as soon as it finds one match (efficient for large datasets). IN evaluates the entire subquery first. For correlated subqueries, EXISTS is generally faster.
Medium

Set Operations

Combine results of multiple SELECT queries. All queries must have the same number of columns with compatible data types.

Operator Returns Duplicates?
UNION All rows from both queries Removes duplicates
UNION ALL All rows from both queries Keeps duplicates (faster)
INTERSECT Only rows in BOTH queries Removes duplicates
EXCEPT / MINUS Rows in first, NOT in second Removes duplicates
SQL
-- UNION: combine and deduplicate
SELECT name FROM employees_india
UNION
SELECT name FROM employees_usa;

-- UNION ALL: faster, keeps duplicates
SELECT product_id, quantity FROM warehouse_a
UNION ALL
SELECT product_id, quantity FROM warehouse_b;

-- INTERSECT: customers who bought both products
SELECT customer_id FROM orders WHERE product_id = 1
INTERSECT
SELECT customer_id FROM orders WHERE product_id = 2;

-- EXCEPT: customers who bought product 1 but NOT 2
SELECT customer_id FROM orders WHERE product_id = 1
EXCEPT
SELECT customer_id FROM orders WHERE product_id = 2;
⚠️ MySQL does not support INTERSECT or EXCEPT natively. Use JOIN or NOT IN as alternatives.

CASE Expressions & Conditional Logic

CASE is SQL's if-then-else. It can appear in SELECT, WHERE, ORDER BY, and aggregate functions.

Searched CASE

SQL
SELECT
    name,
    salary,
    CASE
        WHEN salary >= 80000 THEN 'Senior'
        WHEN salary >= 50000 THEN 'Mid-level'
        WHEN salary >= 30000 THEN 'Junior'
        ELSE 'Intern'
    END AS level
FROM employees;

Simple CASE

SQL
SELECT
    dept_id,
    CASE dept_id
        WHEN 1 THEN 'HR'
        WHEN 2 THEN 'Engineering'
        WHEN 3 THEN 'Sales'
        ELSE 'Other'
    END AS dept_name
FROM employees;

CASE in Aggregates (Pivot Pattern)

SQL — Conditional Aggregation
-- Count employees by salary band in one row per department
SELECT
    dept_id,
    COUNT(CASE WHEN salary >= 80000 THEN 1 END) AS senior_count,
    COUNT(CASE WHEN salary BETWEEN 50000 AND 79999 THEN 1 END) AS mid_count,
    SUM(CASE WHEN gender = 'F' THEN salary ELSE 0 END) AS female_payroll
FROM employees
GROUP BY dept_id;

Views & Indexes

Views

A view is a virtual table based on a SELECT query. It does not store data itself (unlike materialized views).

SQL
-- Create a view
CREATE VIEW high_earners AS
SELECT emp_id, name, salary, dept_id
FROM employees
WHERE salary > 75000;

-- Use like a table
SELECT * FROM high_earners WHERE dept_id = 2;

-- Modify the view definition
CREATE OR REPLACE VIEW high_earners AS
SELECT * FROM employees WHERE salary > 80000;

-- Drop the view
DROP VIEW high_earners;
Aspect View Materialized View
Data stored? No (virtual) Yes (physically stored)
Always up-to-date? Yes (re-runs query) No (needs refresh)
Performance Slower (query reruns) Faster (pre-computed)
Support All RDBMS PostgreSQL, Oracle

Indexes

An index is a data structure (typically a B-Tree) that speeds up SELECT queries at the cost of slower writes (INSERT/UPDATE/DELETE) and more storage.

SQL
-- Create a single-column index
CREATE INDEX idx_emp_salary ON employees(salary);

-- Create a unique index
CREATE UNIQUE INDEX idx_emp_email ON employees(email);

-- Composite index (covers multi-column WHERE)
CREATE INDEX idx_dept_salary ON employees(dept_id, salary);

-- Drop an index
DROP INDEX idx_emp_salary ON employees;
Index Type Description
Primary Index Auto-created on Primary Key, always unique
Unique Index Ensures uniqueness, allows one NULL
Composite Index Covers multiple columns — order matters!
Full-Text Index For text search (MATCH...AGAINST)
Clustered Index Data rows stored in index order (one per table)
Non-Clustered Index Separate index structure pointing to rows
Medium

Normalization

Normalization organizes tables to minimize redundancy and dependency. It is one of the highest-frequency database theory topics in placements.

Why Normalize?

  • Eliminate data redundancy (same data in multiple places)
  • Prevent update anomalies (update in one place, miss others)
  • Prevent insertion anomalies (can't add data without unrelated data)
  • Prevent deletion anomalies (deleting a row loses other data)

First Normal Form (1NF)

Rules: Each column must have atomic (indivisible) values. No repeating groups. Each row must be unique (has a primary key).

❌ Violates 1NF: A column "phones" containing "9876543210, 9988776655" (two values in one cell)
✅ Satisfies 1NF: Create a separate row or table for each phone number

Second Normal Form (2NF)

Must be in 1NF. No partial dependency — every non-key attribute must depend on the entire composite primary key (applies only when PK is composite).

Example
-- Violates 2NF: StudentName depends only on StudentID (partial dependency)
Table: Enrollment(StudentID, CourseID, StudentName, CourseName, Grade)
-- StudentName depends only on StudentID, not on the full PK (StudentID, CourseID)

-- Fixed: split into three tables
Student(StudentID, StudentName)
Course(CourseID, CourseName)
Enrollment(StudentID, CourseID, Grade)

Third Normal Form (3NF)

Must be in 2NF. No transitive dependency — non-key attributes must not depend on other non-key attributes.

Example
-- Violates 3NF: DeptCity depends on DeptID (non-key → non-key)
Table: Employee(EmpID, Name, DeptID, DeptCity)
-- DeptCity is transitively dependent: EmpID → DeptID → DeptCity

-- Fixed:
Employee(EmpID, Name, DeptID)
Department(DeptID, DeptCity)

Boyce-Codd Normal Form (BCNF)

Must be in 3NF. For every functional dependency X → Y, X must be a superkey. BCNF is stricter than 3NF.

Normal Form Requirement (in addition to previous)
1NF Atomic values, unique rows, no repeating groups
2NF No partial dependencies (on composite PK)
3NF No transitive dependencies (non-key → non-key)
BCNF Every determinant must be a superkey
4NF No multi-valued dependencies
💡 Denormalization is intentionally adding redundancy back to improve read performance (used in data warehouses and reporting systems).
Advanced

Window Functions

Window functions perform calculations across a set of table rows related to the current row — without collapsing them into groups like GROUP BY does. They are among the most powerful and commonly tested advanced SQL features.

Syntax

SQL
function_name() OVER (
    PARTITION BY column1       -- divide rows into groups (optional)
    ORDER BY     column2       -- sort within each group (optional)
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  -- frame (optional)
)

Ranking Functions

SQL
SELECT
    name, dept_id, salary,
    ROW_NUMBER()   OVER (PARTITION BY dept_id ORDER BY salary DESC) AS row_num,
    RANK()         OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rank_num,
    DENSE_RANK()   OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dense_rnk,
    NTILE(4)       OVER (ORDER BY salary DESC)                      AS quartile
FROM employees;
Function Tie Handling Example: salaries 100,100,90
ROW_NUMBER() Arbitrary unique numbers 1, 2, 3
RANK() Ties get same rank, gaps after 1, 1, 3
DENSE_RANK() Ties get same rank, no gaps 1, 1, 2
NTILE(n) Distributes into n buckets Depends on n

Practical: Top N per Group

SQL — Top 2 earners per department
SELECT * FROM (
    SELECT
        name, dept_id, salary,
        DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rnk
    FROM employees
) ranked
WHERE rnk <= 2;

LAG and LEAD

SQL
SELECT
    month, revenue,
    LAG(revenue, 1, 0) OVER (ORDER BY month) AS prev_month,
    LEAD(revenue, 1, 0) OVER (ORDER BY month) AS next_month,
    revenue - LAG(revenue) OVER (ORDER BY month) AS mom_growth
FROM monthly_revenue;
-- LAG(col, n, default): value from n rows behind
-- LEAD(col, n, default): value from n rows ahead

Running Totals & Moving Averages (ROWS BETWEEN)

SQL
SELECT
    date, amount,
    SUM(amount) OVER (
        ORDER BY date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total,
    AVG(amount) OVER (
        ORDER BY date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS seven_day_avg
FROM daily_sales;

FIRST_VALUE / LAST_VALUE

SQL
SELECT
    name, dept_id, salary,
    FIRST_VALUE(name) OVER (
        PARTITION BY dept_id ORDER BY salary DESC
    ) AS top_earner_in_dept
FROM employees;

CTEs & Recursive Queries

Common Table Expressions (CTEs) create named, temporary result sets that improve readability and enable recursion.

Basic CTE

SQL
-- WITH clause defines the CTE
WITH dept_averages AS (
    SELECT dept_id, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY dept_id
)
SELECT e.name, e.salary, d.avg_sal
FROM employees e
JOIN dept_averages d ON e.dept_id = d.dept_id
WHERE e.salary > d.avg_sal;

Multiple CTEs

SQL
WITH
high_sal_emps AS (
    SELECT * FROM employees WHERE salary > 70000
),
dept_info AS (
    SELECT * FROM departments WHERE location = 'Delhi'
)
SELECT h.name, d.dept_name
FROM high_sal_emps h
JOIN dept_info d ON h.dept_id = d.dept_id;

Recursive CTE

A recursive CTE references itself. Used for hierarchical data (org charts, file systems, bill of materials).

SQL — Org Chart Traversal
WITH RECURSIVE org_chart AS (
    -- Base case: start from the CEO (no manager)
    SELECT emp_id, name, manager_id, 0 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive case: join employees to their managers
    SELECT e.emp_id, e.name, e.manager_id, oc.level + 1
    FROM employees e
    INNER JOIN org_chart oc ON e.manager_id = oc.emp_id
)
SELECT * FROM org_chart ORDER BY level, name;
💡 Recursive CTEs always have: (1) an anchor/base query, (2) UNION ALL, (3) a recursive query that references the CTE name, (4) a termination condition (implicit via JOIN).

Stored Procedures & User-Defined Functions

Stored Procedure

A stored procedure is a precompiled, named block of SQL code stored on the server. It can accept parameters, execute logic, and return results.

SQL — MySQL Stored Procedure
DELIMITER //

CREATE PROCEDURE GetEmployeesByDept(
    IN  p_dept_id  INT,
    OUT p_count    INT
)
BEGIN
    SELECT name, salary
    FROM employees
    WHERE dept_id = p_dept_id
    ORDER BY salary DESC;

    SELECT COUNT(*) INTO p_count
    FROM employees
    WHERE dept_id = p_dept_id;
END //

DELIMITER ;

-- Call it:
CALL GetEmployeesByDept(2, @total);
SELECT @total;

User-Defined Function (UDF)

Unlike stored procedures, functions must return a value and can be used inside SELECT statements.

SQL
DELIMITER //

CREATE FUNCTION get_annual_salary(monthly_sal DECIMAL(10,2))
RETURNS DECIMAL(12,2)
DETERMINISTIC
BEGIN
    RETURN monthly_sal * 12;
END //

DELIMITER ;

-- Use in SELECT:
SELECT name, get_annual_salary(salary) AS annual
FROM employees;
Aspect Stored Procedure Function
Returns value? Optional (via OUT params) Mandatory (single value)
Use in SELECT? No Yes
Exceptions/Transactions Yes Limited
Called with CALL proc_name() SELECT func_name()

Triggers

A trigger is a stored program that automatically executes in response to a DML event (INSERT, UPDATE, DELETE) on a table.

Timing Events Description
BEFORE INSERT / UPDATE / DELETE Runs before the DML operation
AFTER INSERT / UPDATE / DELETE Runs after the DML operation
SQL — Audit Trigger
DELIMITER //

CREATE TRIGGER log_salary_change
AFTER UPDATE ON employees
FOR EACH ROW
BEGIN
    IF OLD.salary != NEW.salary THEN
        INSERT INTO salary_audit (emp_id, old_salary, new_salary, changed_at)
        VALUES (NEW.emp_id, OLD.salary, NEW.salary, NOW());
    END IF;
END //

DELIMITER ;
-- OLD: row values before the operation
-- NEW: row values after the operation (not available in DELETE)
SQL — BEFORE INSERT validation trigger
DELIMITER //
CREATE TRIGGER validate_salary
BEFORE INSERT ON employees
FOR EACH ROW
BEGIN
    IF NEW.salary < 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Salary cannot be negative';
    END IF;
END //
DELIMITER ;

Transactions & ACID Properties

Transactions group SQL statements into a single logical unit. ACID properties are fundamental database theory — almost always asked in placement interviews.

ACID Properties

Property Meaning Example
Atomicity All or nothing — all operations succeed or all are rolled back Bank transfer: debit AND credit both happen, or neither does
Consistency DB must go from one valid state to another — all constraints maintained Balance can't go negative if constraint says no
Isolation Concurrent transactions execute as if sequential Two users updating same account don't interfere
Durability Committed changes survive system failures Data written to disk even after a crash

TCL Commands

SQL — Bank Transfer Transaction
START TRANSACTION;

UPDATE accounts SET balance = balance - 5000
WHERE account_id = 101;   -- debit

UPDATE accounts SET balance = balance + 5000
WHERE account_id = 202;   -- credit

-- If everything is OK:
COMMIT;

-- If something went wrong:
ROLLBACK;
SQL — SAVEPOINT
START TRANSACTION;
INSERT INTO orders VALUES (1001, ...);
SAVEPOINT sp1;   -- mark a checkpoint

INSERT INTO order_items VALUES (...);
ROLLBACK TO sp1;  -- undo back to savepoint (order_items rolled back)
COMMIT;            -- commits only the orders insert

Transaction Isolation Levels

Level Dirty Read Non-Repeatable Read Phantom Read
READ UNCOMMITTED Possible Possible Possible
READ COMMITTED Not possible Possible Possible
REPEATABLE READ Not possible Not possible Possible
SERIALIZABLE Not possible Not possible Not possible
💡 Dirty Read: Reading uncommitted data | Non-Repeatable Read: Same query returns different data in same transaction | Phantom Read: New rows appear between two reads

Query Optimization

Understanding how to write efficient SQL and analyze query performance is critical for advanced placement interviews.

EXPLAIN / EXPLAIN ANALYZE

SQL
-- Shows the query execution plan
EXPLAIN SELECT * FROM employees WHERE salary > 50000;

-- Shows actual execution stats (PostgreSQL / MySQL 8+)
EXPLAIN ANALYZE SELECT * FROM employees WHERE salary > 50000;

Key EXPLAIN output fields to know:

Field Meaning
type Access method: ALL (full scan) → range → ref → eq_ref → const (best to worst → best)
possible_keys Indexes that could be used
key Index actually used (NULL = no index)
rows Estimated rows to examine
Extra "Using index" = covering index (fast); "Using filesort" = bad sign

Optimization Best Practices

Tips
-- 1. Use indexes on WHERE, JOIN, and ORDER BY columns
CREATE INDEX idx_salary ON employees(salary);

-- 2. Avoid functions on indexed columns in WHERE (breaks index)
-- BAD:
WHERE YEAR(hire_date) = 2023
-- GOOD:
WHERE hire_date BETWEEN '2023-01-01' AND '2023-12-31'

-- 3. SELECT only needed columns
-- BAD:
SELECT * FROM employees
-- GOOD:
SELECT emp_id, name, salary FROM employees

-- 4. Use EXISTS instead of IN for large subqueries
-- 5. Avoid DISTINCT when unnecessary
-- 6. Use LIMIT to cap large result sets
-- 7. Prefer JOINs over correlated subqueries for performance
-- 8. Avoid leading wildcards in LIKE
WHERE name LIKE '%kumar'  -- BAD: can't use index
WHERE name LIKE 'kumar%'  -- GOOD: can use index

Top Placement Interview Questions

Classic SQL problems that appear repeatedly in placement tests at top companies. Understand the pattern, not just the answer.

Q1: Find the Nth highest salary

SQL — Multiple approaches
-- Using LIMIT OFFSET (MySQL)
SELECT DISTINCT salary FROM employees
ORDER BY salary DESC
LIMIT 1 OFFSET N-1;  -- replace N with the desired rank

-- Using subquery (any DB)
SELECT MAX(salary) FROM employees
WHERE salary < (
    SELECT MAX(salary) FROM employees
);  -- This gives the 2nd highest

-- Using DENSE_RANK (most robust)
SELECT salary FROM (
    SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
    FROM employees
) t
WHERE rnk = N;  -- replace N

Q2: Find duplicate records

SQL
-- Find duplicate emails
SELECT email, COUNT(*) AS cnt
FROM employees
GROUP BY email
HAVING COUNT(*) > 1;

-- Delete duplicates, keep one row
DELETE FROM employees
WHERE emp_id NOT IN (
    SELECT MIN(emp_id)
    FROM employees
    GROUP BY email
);

Q3: Employees earning more than their manager

SQL
SELECT e.name AS employee, m.name AS manager
FROM employees e
JOIN employees m ON e.manager_id = m.emp_id
WHERE e.salary > m.salary;

Q4: Consecutive seat numbers / streak detection

SQL
-- Find students sitting in consecutive seats
SELECT DISTINCT a.seat_id
FROM cinema a
JOIN cinema b
  ON ABS(a.seat_id - b.seat_id) = 1
WHERE a.free = 1 AND b.free = 1
ORDER BY a.seat_id;

Q5: Running total / cumulative sum

SQL
SELECT
    date, amount,
    SUM(amount) OVER (ORDER BY date) AS cumulative_total
FROM transactions;

Q6: Find customers who placed orders every month

SQL
SELECT customer_id
FROM orders
GROUP BY customer_id
HAVING COUNT(DISTINCT DATE_FORMAT(order_date, '%Y-%m')) = (
    SELECT COUNT(DISTINCT DATE_FORMAT(order_date, '%Y-%m'))
    FROM orders
);

Q7: Pivot — rows to columns

SQL — Conditional Aggregation Pivot
-- Convert rows of (dept, month, sales) to columns per month
SELECT
    dept_id,
    SUM(CASE WHEN month = 'Jan' THEN sales ELSE 0 END) AS Jan,
    SUM(CASE WHEN month = 'Feb' THEN sales ELSE 0 END) AS Feb,
    SUM(CASE WHEN month = 'Mar' THEN sales ELSE 0 END) AS Mar
FROM dept_sales
GROUP BY dept_id;

Quick Reference: Must-Know Concepts

Topic Key Point to Remember
RANK vs DENSE_RANK RANK has gaps; DENSE_RANK doesn't
DELETE vs TRUNCATE DELETE is DML (rollback-able); TRUNCATE is DDL
WHERE vs HAVING WHERE filters rows; HAVING filters groups
UNION vs UNION ALL UNION removes duplicates; UNION ALL keeps them (faster)
INNER vs LEFT JOIN INNER = only matches; LEFT = all left + matches
NULL comparisons Use IS NULL/IS NOT NULL, never = NULL
Execution order FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY→LIMIT
Index trade-off Faster reads, slower writes; don't over-index
Normalization 3NF No transitive dependencies (A→B→C, fix by separating)
ACID Atomicity, Consistency, Isolation, Durability
